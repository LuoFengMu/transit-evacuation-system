# 工程版本迭代总览

本文档说明"城市大客流与突发事件的公交-轨道协同疏散仿真与调度优化系统"的工程版本划分。版本划分遵循从最小闭环到完整系统的思路，每个版本都是可独立运行的完整系统。

**设计原则**：
- Web 展示从 v0.1.0 起步，每版增强，不推迟到末期
- v0.2.0 加入简化仿真验证，避免调度方案无法验证
- 每个版本解决一个主问题，避免同时引入过多新组件

## 版本路线

| 版本 | 阶段名称 | 核心目标 | 新增模块 |
| --- | --- | --- | --- |
| v0.1.0 | 最小原型 + 基础地图 | 路网、事件、需求点、最短路径 + Streamlit 地图展示 | OSMnx、NetworkX、Streamlit |
| v0.2.0 | 公交调度 + 简化仿真 | OR-Tools 调度优化 + 简化离散事件仿真验证 | OR-Tools、车辆/depot 管理、简化仿真引擎 |
| v0.3.0 | SUMO 动态仿真 | SUMO 动态仿真、道路封闭、动态拥堵 + FastAPI 后端 | SUMO/TraCI、FastAPI、路网转换 |
| v0.4.0 | 公交-轨道协同 | 轨道站点容量、线路运能、接驳分流、方案对比 | 轨道数据、协同优化、站点压力评估 |
| v0.5.0 | 完整 Web 可视化 | 交互式场景配置、仿真看板、指标仪表板 | 控制面板、轨迹回放、报告导出 |
| v1.0.0 | 论文答辩稳定版 | 固化代码/数据/场景/环境，完整文档 | Docker 环境、实验报告、论文图表 |

## 版本间依赖关系

```text
v0.1.0 最小原型 + 基础地图
    │ Streamlit 起步
    ↓
v0.2.0 公交调度 + 简化仿真
    │ OR-Tools + 简化仿真验证
    ↓
v0.3.0 SUMO 动态仿真
    │ 完整 SUMO + FastAPI 后端
    ↓
v0.4.0 公交-轨道协同
    │ 轨道容量 + 接驳分流
    ↓
v0.5.0 完整 Web 可视化
    │ 交互增强 + 报告导出
    ↓
v1.0.0 论文答辩稳定版本
    固化一切，确保可复现
```

## 推荐开发节奏

1. 每个版本只解决一个主问题。
2. 每个版本都应保留可运行入口、配置文件、样例数据和结果报告。
3. 每个版本都应能追溯到代码版本、数据版本、场景版本、算法参数和实验结果（记录在 `simulation_run` 表）。
4. `main` 分支只保存可演示版本，`dev` 分支用于日常开发。
5. 每个版本完成后打 Git tag，并在 `docs/release_notes/` 中保存 Release Note。

## 每个版本文档的固定结构

- 版本目标
- 功能范围
- 数据需求
- 算法与模型
- 仿真与可视化
- 目录与文件交付
- 验收标准
- 已知限制
- 下一版本衔接

## 版本资料保存

每个版本建立独立实验目录：

```text
experiments/
├── v0.1_minimum_prototype/
├── v0.2_bus_dispatch/
├── v0.3_sumo_simulation/
├── v0.4_bus_rail_cooperation/
├── v0.5_web_dashboard/
└── v1.0_final_release/
```

每个目录至少保存：`config.yaml`、`data_manifest.json`、`simulation_run.json`、`run.log`、`metrics.csv`、`figures/`、`report.md`
