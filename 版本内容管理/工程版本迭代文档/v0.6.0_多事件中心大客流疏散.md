# v0.6.0 多事件中心大客流疏散

## 版本目标

v0.6.0 的目标是将当前单事件中心大客流疏散模型，升级为多事件中心并发疏散模型。系统应支持多个大型活动、交通枢纽或商圈同时出现大客流，统一进行需求生成、公交调度、轨道协同和评价分析。

本版本继续沿用 v0.5.0 已完成的 Web / CLI / Batch / manifest / summary.csv 实验底座，不引入 PostGIS、FastAPI、JuPedSim 或 MATSim 作为主线依赖。

## 功能范围

- 支持多事件 YAML：

```yaml
events:
  - event_id: pengcheng_square
    location: 彭城广场
    center: [117.205, 34.268]
    radius_m: 1500
    demand_scale: 30000
    priority: 1

  - event_id: xuzhou_east_station
    location: 徐州东站
    center: [117.312, 34.267]
    radius_m: 1200
    demand_scale: 20000
    priority: 2
```

- 每个事件独立生成/平移/缩放需求点，并合并为全局 `demand_gdf`
- `demand_gdf` 增加 `event_id`、`event_location`、`event_priority`
- 道路影响区支持多个事件中心
- 公交车辆面对全局需求统一调度
- 轨道站容量在多个事件之间共享和竞争
- 输出全局指标与分事件指标
- Streamlit 地图按事件颜色区分需求点与影响区
- CLI / Batch 支持多事件场景文件

## 建议实施顺序

### 6.1 多事件配置与数据结构

- 新增多事件场景：`configs/scenarios/M_multi_event.yaml`
- 支持 `event` 单对象和 `events` 列表两种格式，保持向后兼容
- 新增事件解析工具，将单事件和多事件统一为 `EventSpec` 列表

### 6.2 多事件需求生成与预处理

- 每个事件独立生成或平移需求点
- 对每个事件应用道路吸附、水体过滤和需求缩放
- 合并后保留 `event_id`
- 验证全局人数 = 各事件人数之和

### 6.3 多事件影响区与路径计算

- `get_affected_roads` 支持多事件中心
- 输出 `affected_by_event` 结构，记录每个事件影响的道路
- 若影响区重叠，保留多事件标签

### 6.4 全局调度与轨道协同

- 公交调度面对全局需求统一求解
- 可按 `priority` 调整服务顺序或未服务惩罚
- 轨道协同分配时，站点容量对所有事件共享
- 输出每个事件的完成率、轨道分担率、公交分担率、未服务人数

### 6.5 可视化与批量实验

- Streamlit 地图按事件着色
- CLI 显示多事件摘要
- Batch 的 `summary.csv` 增加：
  - `n_events`
  - `event_ids`
  - `global_completion_rate`
  - `event_{id}_completion_rate`
  - `event_{id}_unserved`

## 验收标准

- 单事件 YAML 仍可正常运行，结果不回退
- 多事件 YAML 可通过 Streamlit、CLI、Batch 三入口运行
- 每个需求点都有 `event_id`
- 多事件需求人数守恒
- 输出全局指标和分事件指标
- `summary.csv` 可记录多事件实验结果
- 新增测试覆盖：
  - 多事件 YAML 解析
  - 多事件需求合并
  - 分事件指标守恒
  - 多事件 batch 输出

## 暂缓事项

- PostGIS：等多事件实验数据规模变大、空间查询需求明确后再接入
- JuPedSim：等识别出高压轨道站后，作为站内微观疏散增强
- MATSim：远期城市级多智能体扩展，不进入 v0.6.0 主线
- FastAPI/Docker：除非近期需要部署或多人使用，否则继续后移

## 下一版本衔接

v0.7.0+ 可在多事件模型稳定后，按需引入真实人口/公交/轨道数据、PostGIS 空间数据库、站内 JuPedSim 仿真或服务化 API。
