# 发布与运维指南

> 本文档覆盖发布 checklist、版本回滚、兼容性矩阵和日常运维。是《工程实施与版本管理方案》第 12-15 节的操作指南。

---

## 1. 发布 Checklist

每个版本发布前，逐项确认：

### 代码
- [ ] 所有 feature 分支已合并到 `dev`
- [ ] `dev` → `release/vX.Y.Z` 分支已创建
- [ ] 单元测试全部通过：`pytest tests/unit/`
- [ ] 集成测试通过：`pytest tests/integration/`
- [ ] 仿真回归测试通过（关键指标波动 < 5%）
- [ ] 代码无 TODO/FIXME 遗留（或已记录在 Release Note 已知问题中）
- [ ] `VERSION` 文件已更新
- [ ] `CHANGELOG.md` 已更新

### 数据
- [ ] `data/manifest/data_manifest_vX.Y.json` 已创建
- [ ] 所有引用数据文件存在且可读取
- [ ] 大型数据文件（> 10MB）已通过 DVC 管理

### 场景
- [ ] 至少一个场景配置可从头运行到结果
- [ ] 场景 YAML 中 `data_version` 与 manifest 一致

### 文档
- [ ] `docs/release_notes/vX.Y.Z.md` 已填写
- [ ] README 中的安装步骤可在干净环境复现
- [ ] API 文档已更新（如有接口变更）

### 环境
- [ ] `requirements.txt` 版本号已锁定
- [ ] `environment.yml` 已更新
- [ ] `docker-compose.yml` 服务可正常启动
- [ ] `.env.example` 包含所有新增环境变量

### 发布操作

```bash
# 1. 合并到 main
git checkout main
git merge release/v0.4.0

# 2. 打 tag
git tag -a v0.4.0 -m "v0.4.0: 公交-轨道协同疏散版本"

# 3. 推送
git push origin main
git push origin v0.4.0

# 4. 创建 GitHub Release
gh release create v0.4.0 \
  --title "v0.4.0 - 公交-轨道协同疏散" \
  --notes-file docs/release_notes/v0.4.0.md
```

## 2. 版本回滚

### 2.1 确定回滚目标

查看可回退的版本：

```bash
git tag --list 'v*' --sort=-version:refname
```

查看目标版本的 `simulation_run` 记录以确定完整的版本依赖：

```bash
# 如果该版本的实验数据仍在
cat experiments/v0.4_*/simulation_run.json
```

### 2.2 执行回滚

```bash
# 1. 代码回滚
git checkout v0.3.0
# 或创建 hotfix 分支：
git checkout -b hotfix/rollback-to-v0.3.0 v0.3.0

# 2. 数据回滚
# 从 data/manifest/ 找到目标版本的 manifest
cp data/manifest/data_manifest_v0.3.json data/manifest/
# 恢复 DVC 数据文件
dvc checkout data_manifest_v0.3.json

# 3. 环境回滚
git checkout v0.3.0 -- requirements.txt environment.yml docker-compose.yml
conda env update -f environment.yml
docker compose up -d

# 4. 验证
pytest tests/
python run_experiment.py --config configs/scenarios/scenario_v0.3_baseline.yaml
```

### 2.3 回滚记录

在 `CHANGELOG.md` 中记录：

```markdown
## [回滚] 2026-05-12
- 从 v0.4.0 回滚到 v0.3.0
- 原因：轨道协同模型求解不稳定，需修订约束建模
- 操作人：xxx
```

### 3.2 已知冲突

- **SUMO 1.19 Python 工具** 需要 Python 3.9-3.11，不兼容 Python 3.12+
- **OR-Tools 9.10+** 支持 Python 3.12，但与 SUMO 的版本要求冲突——建议固定 Python 3.11
- **GeoPandas 0.14+** 依赖 GEOS 3.12+，通过 conda 安装可自动解决依赖
- **OpenTripPlanner 2.x** 是 Java 17 应用，通过 Docker 运行避免环境冲突

## 4. 日常运维

### 4.1 新增功能的标准流程

```bash
git checkout dev
git checkout -b feature/my-new-feature
# 开发 + 测试
pytest tests/
# 合并
git checkout dev
git merge feature/my-new-feature
git branch -d feature/my-new-feature
```

### 4.2 紧急修复流程

```bash
git checkout main
git checkout -b hotfix/fix-sumo-route-export
# 修复 + 测试
pytest tests/
# 合并到 main 和 dev
git checkout main
git merge hotfix/fix-sumo-route-export
git tag -a v0.4.1 -m "Hotfix: 修复 SUMO route 导出错误"
git checkout dev
git merge hotfix/fix-sumo-route-export
git branch -d hotfix/fix-sumo-route-export
```

### 4.3 数据的日常备份

```bash
# PostGIS 数据库备份
pg_dump transit_evacuation > backups/db_$(date +%Y%m%d).sql

# DVC 数据同步
dvc push

# 实验数据归档
tar -czf backups/experiments_$(date +%Y%m%d).tar.gz experiments/
```

### 4.4 日志监控

仿真运行日志位置：
- FastAPI: Docker logs 或 `outputs/logs/api.log`
- SUMO: `outputs/sumo/` 和 `outputs/logs/sumo.log`
- 调度优化: `outputs/logs/dispatch.log`

日志级别建议：开发环境 `DEBUG`，演示/答辩环境 `INFO`。

## 6. 故障排查

| 症状 | 可能原因 | 排查步骤 |
| --- | --- | --- |
| SUMO 启动失败 | 路网转换错误 | 检查 `sumo/networks/*.net.xml` 是否生成成功 |
| OR-Tools 无解 | 约束过紧或需求 > 容量 | 检查 `run.log` 中的 solver 状态，尝试放宽约束 |
| 仿真结果异常波动 | random_seed 不一致 | 确认 `config.yaml` 中 `random_seed` 非空 |
| Web 页面无数据 | FastAPI 未启动或端口冲突 | 检查 `docker compose ps`，确认 8000/8501 端口 |
| PostGIS 连接失败 | 数据库未初始化 | `docker compose logs postgis`，检查环境变量 |
