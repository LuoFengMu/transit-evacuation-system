- `walk_direct_share = (walk_self + walk_rail) / total_demand`
- `total_evacuated = walk_self + walk_rail + bus_rail + bus_periphery`
- `unserved = allocation_result.unassigned_people`

- `streamlit_app.py`：侧边栏新增「成本矩阵模式」下拉框（默认 YAML 取值），导入统一函数，条件调用

### 3.2 详情 — 步行网络接入升级

修改文件：
- `src/walking/access.py`：新增 `snap_to_nearest_network()` 使用 OSM walk graph + Dijkstra 计算真实步行距离；`compute_access_matrix()` 接受 `mode` / `walk_G` 参数；新增 `WALK_EUCLIDEAN` / `WALK_NETWORK` 模式常量
- `streamlit_app.py`：侧边栏轨道参数区新增「步行距离模式」radio（欧氏距离/步行网络），`walk_network` 模式下自动加载 OSM walk graph (28k节点/84k边) 并传入 `compute_access_matrix`
- `configs/scenarios/A/B/C_*.yaml`：`rail.walk_mode: "euclidean_fast"`
- `tests/test_walk_access.py`：7 测试（欧氏模式结构、无效模式报错、网络模式需要 graph、mock Dijkstra 网络模式、步行时间计算）

烟测：`walk_network` 模式在真实徐州 walk graph 上通过：D0→S0 距离 901m/时间 649s。

### 3.3 详情 — 需求点道路节点采样 + 水体过滤

修改文件：
- `src/demand/generator.py`：新增 `snap_to_road_nodes()` 将需求点吸附到最近道路节点（全局模式，不再仅 >200m）；新增 `filter_water_bodies()` 使用 OSM 水体数据 + sjoin 检测并迁移位于水域的需求点（缓存至 `cache/water_polygons.geojson`）
- `streamlit_app.py`：侧边栏新增「需求点预处理」expander，含「吸附到道路节点」（默认开启）和「水体过滤」（默认关闭）两个 checkbox；替换原有仅 >200m 的简易吸附逻辑
- `configs/scenarios/A/B/C_*.yaml`：`demand.snap_to_road: true` / `demand.filter_water: false`
- `tests/test_demand_snap.py`：9 测试（吸附返回新 GDF、保留 people_count/demand_id、水体过滤搬迁/非水体不变/空水域无操作、人数保留、**最近节点在水体内时跳过到次近陆地节点**）

修复：`_nearest_non_water_node()` 搜索最多 20 个候选节点，使用 `water_union.contains(pt)` 排除水体覆盖的节点。

### 3.4 详情 — 轨道容量敏感性分析

修改文件：
- `src/rail/cooperative.py`：`allocate_cooperative` 新增 `capacity_factor: float = 1.0` 参数，乘以 `dynamic_capacity_pax_h` 计算有效容量；`station_caps` = `int(dynamic_capacity_pax_h * capacity_factor * time_window_h)`
- `streamlit_app.py`：侧边栏轨道参数区新增 `select_slider`「轨道容量情景」（保守 ×0.7 / 基准 ×1.0 / 乐观 ×1.2）+「容量敏感性分析」checkbox；运行三组因子并输出对比表（完成率/轨道分担/未服务/过载站数）
- `configs/scenarios/B/C_*.yaml`：`rail.capacity_factor: 1.0`
- `tests/test_rail_sensitivity.py`：6 测试（保守≤基准、乐观≥基准、factor 缩放有效容量、默认 1.0、**压力因子断面：同一 arrivals 下保守 > 基准 > 乐观**、人数守恒）

修复：`compute_pressure()` 新增 `capacity_factor` 参数作用于 `capacity_used = dynamic_capacity_pax_h * capacity_factor * time_window_h`；`allocate_cooperative` 传递 `capacity_factor`；敏感性分析传入 `station_pressures`。

### 4.4-4.5 目标 — v0.5.0 可复现实验平台

v0.5.0 不再定位为“继续完善 Web 可视化”，而是优先把已经拆出的主流程变成可脚本化、可批量、可追溯的实验平台。

计划交付：

- `src/app/cli.py`：单场景命令行入口，支持 `--scenario`、`--seed`、`--demand`、`--capacity-factor` 等参数（✅ 已完成）
- `src/app/batch.py` 或 `scripts/run_experiments.py`：批量运行 A/B/C 场景、多 seed、多轨道容量情景
- `outputs/experiments/{experiment_id}/summary.csv`：多次运行的核心指标汇总
- `manifest.json`：记录场景、seed、容量系数、耗时、输出目录和核心指标
- `summary.csv`：每行一次运行，汇总完成率、轨道分担、未服务人数、过载站数、调度状态和耗时
- 集成测试：验证一次最小运行能生成 7 类输出文件，并能读取关键指标

后移事项：

- FastAPI：仅当需要外部系统调用、并发任务管理或前后端分离时再做
- Docker：优先放到 v1.0 环境固化，或在 SUMO/OSMnx 环境复现成为瓶颈时提前
- CI：v0.5.0 有 CLI 和集成测试后再接入，避免 CI 只能跑零散单元测试
