# 项目代码优化与修改建议

## 1. 文档目的

本文基于当前仓库代码与版本文档的审阅结果，整理后续优化和修改方向。重点不再重复技术蓝图，而是回答三个问题：

1. 当前代码哪些地方已经可以支撑演示和论文实验。
2. 哪些实现会影响结果可信度、可复现性或后续扩展。
3. 后续应该按什么优先级修改。

审阅范围包括：

- 主入口：`streamlit_app.py`
- 调度模块：`src/dispatch/`
- SUMO 仿真模块：`src/simulation/`
- 轨道协同模块：`src/rail/`
- 步行接入模块：`src/walking/`
- 评价与报告模块：`src/evaluation/`
- 场景配置：`configs/scenarios/`
- 版本与路线文档：`VERSION`、`CHANGELOG.md`、`版本内容管理/`

## 2. 当前项目状态判断

当前项目已经具备一个较完整的研究原型闭环：

- OSM 路网加载与最短路径计算已经形成基础能力。
- OR-Tools CVRP 公交调度已经接入，并支持需求拆分。
- SUMO headless 仿真、TraCI 道路封闭、轨迹提取和子网裁剪已经有实现。
- 轨道站容量、动态处理能力、站点压力和公交-轨道-步行协同分配已经有核心代码。
- Streamlit 已经集成场景参数、调度、SUMO、轨道协同、方案对比、路径审计和报告导出。

但当前代码更像“演示型研究原型”，还没有完全达到“可复现实验平台”的状态。主要差距集中在：指标口径、复现性、成本矩阵真实性、模块解耦、配置管理、测试与文档一致性。

## 3. 高优先级修改建议

### 3.1 修正评价指标的统计口径

优先级：P0

当前 `src/evaluation/metrics.py` 中，部分指标按“需求点数量”统计，而不是按“人数”统计。例如：

- `rail_share` 只识别 `rail`，但协同分配实际使用的是 `walk_rail` 和 `bus_rail`。
- `bus_direct_share` 只识别 `shelter`，但当前大客流逻辑中使用的是 `bus_periphery`。
- 没有 `dispatch_result` 时，`total_evacuated = total_demand - len(unassigned)`，这里把未分配需求点数量当成了未疏散人数。

影响：

- A/B/C 方案对比中的完成率、轨道分担率、未疏散人数可能失真。
- 论文或答辩中若直接使用这些结果，指标解释风险较高。

建议修改：

1. 在 `AllocationResult` 中增加按方式汇总的人数字段，例如：
   - `mode_people: dict[str, int]`
   - `unassigned_people: int`
   - `served_people: int`
2. `compute_evacuation_metrics()` 统一按人数统计，而不是按需求点个数。
3. 明确方式映射：
   - 轨道分担：`walk_rail + bus_rail`
   - 公交分担：`bus_rail + bus_periphery`
   - 步行分担：`walk_self + walk_rail`
   - 未服务：未被分配的需求点对应人数总和
4. 给该模块补最小单元测试，至少覆盖一个包含 3 个需求点、不同人数、不同方式分配的例子。

### 3.2 统一版本状态

优先级：P0

当前版本信息存在不一致：

- `VERSION` 是 `0.4.0`。
- `streamlit_app.py` 标题是 v0.4.0。
- `src/evaluation/audit.py` docstring 写的是 v0.4.1。
- `技术蓝图与开发路线.md` 中把 v0.4.1 标为当前版本。
- `CHANGELOG.md` 只记录到 v0.3.0。

影响：

- 后续实验报告、答辩截图、版本回溯会混乱。
- 很难判断某次实验对应的是 v0.4.0 还是 v0.4.1。

建议修改：

1. 如果轨迹审计和报告导出已经作为当前稳定能力，建议正式升级为 `0.4.1`。
2. 同步更新：
   - `VERSION`
   - `CHANGELOG.md`
   - `streamlit_app.py` 页面标题和侧边栏版本
   - `版本内容管理/技术蓝图与开发路线.md`
3. 新增一段 v0.4.1 Release Note，说明：
   - 轨迹可信度修复
   - 路径审计
   - 报告导出
   - 当前已知限制

### 3.3 提高实验复现性

优先级：P0

当前 `streamlit_app.py` 中使用 `time.time()` 作为随机种子，对需求规模做正负 15% 扰动。这适合演示，但不适合论文实验。

影响：

- 同一参数下多次运行结果不同。
- 对比实验无法严格复现。
- 仿真指标变化难以判断是算法变化还是随机扰动导致。

建议修改：

1. 在侧边栏增加 `random_seed` 输入框，默认 `42`。
2. 增加“启用需求随机扰动”开关，默认关闭。
3. 场景 YAML 中加入：

```yaml
simulation:
  random_seed: 42

demand:
  random_variation: false
  variation_ratio: 0.15
```

4. 所有随机过程统一从场景配置读取 seed。
5. 输出报告中记录 seed、需求缩放比例和实际总需求。

### 3.4 用路网时间矩阵逐步替代欧氏距离矩阵

优先级：P1

当前调度主流程中仍主要使用 `compute_euclidean_matrix()` 构造公交调度成本。虽然速度快，但对城市路网绕行、桥梁、水体和单行方向不敏感。

影响：

- 调度方案可能在几何上看起来合理，但实际路网通行时间偏差较大。
- 大客流场景下，公交到轨道站或外围疏散点的分配可能被低估。

建议修改：

1. 保留欧氏矩阵作为快速演示模式。
2. 新增“成本矩阵模式”配置：
   - `euclidean_fast`
   - `road_network_time`
   - `cached_network_time`
3. 对常用事件地点、公交站、轨道站预计算缓存。
4. 缓存文件建议保存到：

```text
cache/cost_matrix/
├── xuzhou_event_pengcheng_bus_rail_v0.4.parquet
└── xuzhou_event_olympic_bus_rail_v0.4.parquet
```

5. 对论文实验使用路网时间矩阵，对现场演示使用快速模式。

## 4. 中优先级优化建议

### 4.1 拆分 `streamlit_app.py`

优先级：P1

当前 `streamlit_app.py` 承担了过多职责：页面配置、数据加载、需求平移、需求修正、公交调度、SUMO 仿真、轨道协同、指标计算和展示。随着 v0.5.0 继续增强，单文件会越来越难维护。

建议拆分为：

```text
src/app/
├── config.py              # 路径、默认参数、版本号
├── scenario_state.py      # 侧边栏参数与场景对象
├── data_loader.py         # 路网、需求、公交站、轨道站加载
├── pipeline.py            # 一次分析运行流程
└── tabs.py                # Streamlit tabs 组织
```

短期可以先抽出两类最稳定的代码：

- 路径常量和数据加载函数
- 调度/SUMO/轨道协同 pipeline

### 4.2 修正协同分配中的整点不可拆问题

优先级：P1

`allocate_cooperative()` 目前按需求点整体分配。如果某个需求点人数超过本轮剩余公交运力，可能直接进入下一轮或无法服务，而不是拆分部分人群。

影响：

- 大型活动需求点经缩放后可能达到数千人。
- 整点分配会让公交运力利用率和未服务人数出现跳变。

建议修改：

1. 复用公交调度中的“需求拆分”思想，将大需求点拆为批次。
2. 批次大小可以取：
   - 单车容量
   - 单轮公交容量的一定比例
   - 轨道站动态容量的一定比例
3. `AllocationResult` 同时保留原始需求点汇总结果，避免展示层过碎。

### 4.3 轨道容量模型增加敏感性参数

优先级：P1

当前轨道站容量来自固定数据。真实站厅、站台、闸机、出入口能力不确定，论文中最好体现敏感性分析。

建议新增三组容量系数：

```yaml
rail:
  capacity_scenario: baseline
  capacity_factor:
    conservative: 0.7
    baseline: 1.0
    optimistic: 1.2
```

评价时输出三组结果，体现模型对容量假设的敏感性。

### 4.4 SUMO 出发时间改为稳定散列或显式序号

优先级：P1

`route_builder.py` 中使用 `hash(vid) % 60` 生成车辆出发偏移。Python 的 hash 在不同进程/环境中可能受 hash seed 影响，不适合作为可复现实验依据。

建议修改：

1. 改用车辆序号生成偏移，例如 `vehicle_index % 60`。
2. 或使用稳定散列：

```python
import hashlib
offset = int(hashlib.md5(vid.encode()).hexdigest()[:8], 16) % 60
```

3. 在输出 trip 文件时记录生成参数。

### 4.5 场景 YAML 与主程序真正打通

优先级：P1

仓库中已有 `A_bus_direct.yaml`、`B_rail_priority.yaml`、`C_hybrid_cooperative.yaml`，但 Streamlit 主流程仍大量使用硬编码默认路径、默认事件地点和侧边栏参数。

建议修改：

1. 侧边栏增加“选择场景配置”下拉框。
2. 读取 YAML 后填充默认参数。
3. 运行结果保存时复制场景 YAML 到输出目录。
4. 方案 A/B/C 使用同一个底层 pipeline，通过 `mode` 和开关控制。

## 5. 低优先级但有价值的优化

### 5.1 增加 README

优先级：P2

根目录当前缺少 `README.md`。建议至少包含：

- 项目简介
- 环境安装
- 数据准备
- Streamlit 启动命令
- SUMO 环境说明
- 当前版本能力
- 已知限制

### 5.2 补充最小测试集

优先级：P2

建议先不追求完整覆盖率，而是补 5 个最关键的测试：

```text
tests/
├── test_metrics.py
├── test_rail_capacity.py
├── test_cooperative_allocation.py
├── test_dispatch_split.py
└── test_route_builder_determinism.py
```

重点验证：

- 指标按人数统计
- 轨道压力等级正确
- 协同分配不会超过站点压力阈值
- 需求拆分后人数守恒
- SUMO trip 生成可复现

### 5.3 明确真实数据与模拟数据边界

优先级：P2

当前项目同时包含 OSM 真实路网、Overpass 公交站、手动推算轨道站容量、生成需求点。建议在文档和报告中明确：

- 哪些数据是真实采集。
- 哪些数据是公开资料推算。
- 哪些数据是模拟生成。
- 哪些参数是论文实验假设。

这对答辩时解释系统可信度很重要。

### 5.4 输出实验目录

优先级：P2

建议新增一个轻量级 `run_id` 输出机制，即使暂时不上完整 `run_experiment.py`，也可以让 Streamlit 每次运行保存：

```text
outputs/runs/{run_id}/
├── config.yaml
├── metrics.json
├── report.txt
├── dispatch_summary.csv
├── station_pressure.csv
└── run_meta.json
```

`run_meta.json` 至少记录：

- git commit 或手动版本号
- VERSION
- Python 版本
- SUMO 版本
- random_seed
- 数据文件路径
- 场景配置路径

## 6. 文档体系调整建议

当前文档有两类内容混在一起：

1. 已实现能力：v0.1-v0.4 主链路、轨道协同、SUMO 轨迹审计。
2. 规划能力：FastAPI、PostGIS、Docker、OpenTripPlanner、JuPedSim、MATSim、自动化测试。

建议将文档状态标注清楚：

- `已实现`
- `部分实现`
- `规划中`
- `远期扩展`

尤其是 FastAPI、PostGIS、Docker、run_experiment.py、tests，目前更适合放在 v0.5/v1.0 待办中，不应让读者误以为已经落地。

建议新增或调整以下文档：

```text
README.md
版本内容管理/v0.4.1_当前实现说明.md
版本内容管理/项目代码优化与修改建议.md
版本内容管理/论文实验复现清单.md
```

## 7. 推荐实施顺序

### 第一阶段：先保证结果可信

目标：让当前 v0.4.x 的指标和版本状态可信。

任务：

1. 修正 `compute_evacuation_metrics()` 的人数统计口径。
2. 修正 `rail_share`、`bus_direct_share` 等方式映射。
3. 统一版本号并补 v0.4.1 changelog。
4. 固定随机种子，默认关闭随机扰动。
5. 用稳定规则替代 `hash(vid)` 出发时间。

### 第二阶段：增强复现实验能力

目标：让同一场景可以稳定复现。

任务：

1. 场景 YAML 作为主配置来源。
2. Streamlit 输出 `outputs/runs/{run_id}`。
3. 增加 metrics、rail、allocation、route_builder 的最小测试。
4. 补 README 和当前实现说明。

### 第三阶段：提高模型真实性

目标：让实验更接近真实交通系统。

任务：

1. 增加路网时间矩阵模式和缓存。
2. 步行接入从欧氏距离升级为 `network_type="walk"`。
3. 需求点生成改为道路节点采样，并过滤水体。
4. 轨道容量增加保守/基准/乐观敏感性分析。

### 第四阶段：再做工程架构升级

目标：进入 v0.5 可复现实验平台阶段。

任务：

1. 拆分 `streamlit_app.py`。
2. 新增 CLI 单场景入口，让实验可以脱离 Streamlit 手动点击运行。
3. 新增批量实验入口，支持 A/B/C 场景、多 seed、多容量系数组合运行。
4. 增加实验 manifest 与 `summary.csv`，记录参数、版本、输出目录和核心指标。
5. 补主流程集成测试，验证一次运行能产出完整 7 类输出文件。
6. 最后再按需评估 FastAPI/PostGIS、Docker 和完整 CI。

## 8. 结论

当前项目已经不是空架子，而是一个功能相对完整的疏散仿真研究原型。下一步最值得投入的不是继续堆新组件，而是先把“指标可信、版本一致、实验可复现”三件事做好。

建议近期主线定位为：

> v0.4.1：稳定可信的公交-轨道协同演示版。

之后再进入：

> v0.5.0：可复现实验平台版（CLI、批量实验、manifest、summary.csv）。

这样路线会更稳，也更利于论文实验、答辩展示和后续工程扩展。
