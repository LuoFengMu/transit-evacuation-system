# 城市大客流与突发事件的公交-轨道协同疏散仿真与调度优化系统

## 工程实施与版本管理方案

---

## 1. 建设思路

本系统按照"可落地、可演示、可复现"的工程目标建设，围绕**数据接入、调度优化、仿真执行、可视化展示**四个核心环节，先构建能够完整运行的最小闭环，再逐步扩展。

**核心原则：每个版本都是可运行的完整系统，而非半成品堆积。**

推荐主技术组合：

> **SUMO + OR-Tools + OpenTripPlanner + PostgreSQL/PostGIS + FastAPI/Streamlit**

可选增强模块：

- **JuPedSim**：地铁站、公交枢纽等室内人群疏散仿真（第三阶段）
- **MATSim / pt2matsim**：城市级多智能体疏散仿真（远期扩展）

---

## 2. 技术组合详解

### 2.1 路网与数据层：OSMnx + GTFS + PostGIS

| 组件 | 作用 |
| --- | --- |
| OSMnx | 获取城市道路网络、步行网络，支持基于 OSM 的空间网络分析 |
| GTFS | 表示公交、地铁线路、站点、班次和时刻表 |
| PostgreSQL/PostGIS | 存储路网、站点、疏散点、避难点、车辆 depot、调度结果和仿真结果 |

### 2.2 公交与道路仿真层：SUMO + TraCI

SUMO 作为主仿真引擎，通过 TraCI 接口在线读取/修改车辆、道路、信号状态。负责：道路拥堵演化仿真、公交车辆运行仿真、应急调度方案执行、道路封闭与容量下降模拟。

### 2.3 调度优化层：Google OR-Tools

将应急公交调度抽象为带容量约束、时间窗约束的车辆路径问题（CVRP/VRPTW）。

**模型输入**：疏散需求点、避难点/轨道站点、车辆 depot、车辆容量、道路通行时间矩阵、轨道站点容量。

**模型输出**：每辆应急公交的接驳路径、需求点分配方案、公交发车顺序、总疏散时间和车辆利用率。

### 2.4 公交-地铁换乘路径层：OpenTripPlanner

用于公交、地铁、步行多方式路径规划。第一版可先用 OSMnx/NetworkX 替代，后续升级。

### 2.5 地铁站内部疏散层：JuPedSim（可选增强）

模拟站厅、站台、闸机、楼梯和换乘通道中的行人运动，适合第二阶段或第三阶段引入。

### 2.6 后端与可视化层：FastAPI + Streamlit

- **FastAPI**：后端 API，封装场景管理、调度优化、仿真调用和结果查询。
- **Streamlit**：交互式仿真看板，从 v0.1.0 起逐步搭建，不推迟到末期。

前端界面要素：事件位置选择、影响范围设置、道路封闭设置、疏散需求点设置、车辆参数设置、轨道接驳开关、仿真控制（开始/暂停/重置）、疏散完成率曲线、路网拥堵热力图、公交轨迹和站点压力图。

### 2.7 开源项目定位

| 项目 | 定位 | 引入阶段 |
| --- | --- | --- |
| SUMO | 主仿真引擎 | v0.3.0 |
| TraCI | 在线控制接口 | v0.3.0 |
| OR-Tools | 公交调度与路径优化 | v0.2.0 |
| OpenTripPlanner | 多方式路径规划 | v0.4.0+ |
| OSMnx | 路网获取与空间分析 | v0.1.0 |
| PostGIS | 空间数据存储与查询 | v0.1.0 |
| Streamlit | 演示界面 | v0.1.0（逐步增强） |
| FastAPI | 后端接口服务 | v0.3.0 |
| JuPedSim | 室内疏散仿真 | v0.6.0+ |
| MATSim | 多智能体城市级仿真 | 远期 |

## 3. 系统架构

```text
OSM 路网 / GTFS 公交地铁 / 疏散需求点 / 避难点 / 车辆 depot
        ↓
PostgreSQL + PostGIS
        ↓
场景生成层
突发事件位置、影响范围、道路封闭、需求分布、应急车辆参数
        ↓
路径与调度层
OpenTripPlanner / OSMnx / OR-Tools
        ↓
仿真执行层
SUMO + TraCI
        ↓
可选站内疏散层
JuPedSim
        ↓
评价分析层
疏散时间、完成率、车辆利用率、拥堵变化、站点压力、未疏散人数
        ↓
Web 可视化层
FastAPI + Streamlit
```

### 5.1 v0.1.0：最小原型 + 基础地图

**目标**：加载 OSM 路网，设置突发事件区域，生成疏散需求点和避难点，计算最短路径，在 Streamlit 地图上展示。

**包含模块**：
- OSMnx 路网获取
- 疏散点与避难点设置
- 突发事件区域定义
- NetworkX 最短路径
- Streamlit 基础地图展示（路网、需求点、避难点、路径）

**验收标准**：
- 成功加载目标区域路网
- Web 页面显示路网、事件区域、需求点、避难点
- 每个需求点可计算并显示到避难点的最短路径
- 输出每条路径的距离和预计通行时间

### 5.2 v0.2.0：公交调度 + 简化仿真

**目标**：加入公交车辆和 OR-Tools 调度优化，并通过简化离散事件仿真验证调度方案的可行性。本版本不依赖 SUMO，但提供比纯静态路径更有参考价值的运行时间估计。

**新增模块**：
- 公交车辆和 depot 数据管理
- OR-Tools CVRP/VRPTW 建模
- 简化离散事件仿真（排队模型 + 路段通行时间 + 容量约束）
- 需求点分配与车辆路径输出
- Streamlit 页面增加调度结果展示

**关键变更**：v0.2.0 包含一个轻量级仿真组件，用排队论和路段容量模型近似公交运行，使得调度方案的"总疏散时间"不是纯粹的路网最短路径求和，而是考虑了车辆容量、需求点排队和路段拥堵的近似结果。这为 v0.3.0 接入 SUMO 提供了对照基线。

**验收标准**：
- OR-Tools 输出可行调度方案
- 简化仿真能给出近似的总疏散时间和车辆利用率
- 可在 Web 页面查看车辆路径和服务序列
- 静态最短路径估计 vs 简化仿真结果可对比

### 5.3 v0.3.0：SUMO 动态仿真

**目标**：将 OSM 路网转为 SUMO 网络，调度方案转为 SUMO route 文件，通过 TraCI 控制仿真，记录动态拥堵和公交运行。搭建 FastAPI 后端接口。

**新增模块**：
- OSM → SUMO 路网转换
- OR-Tools 输出 → SUMO route 文件生成
- TraCI 动态控制（道路封闭、路径调整）
- 拥堵状态记录与车辆轨迹输出
- FastAPI 后端（仿真任务管理、结果查询）

**验收标准**：
- OSM 路网可成功转为 SUMO 网络
- 调度方案生成 SUMO route 文件并正常仿真
- 至少一种道路封闭场景可模拟
- FastAPI 接口可查询仿真状态和结果
- 输出公交运行时间、拥堵指标

### 5.4 v0.4.0：公交-轨道协同

**目标**：将轨道交通纳入疏散体系，地铁站作为具有容量和运能约束的大容量疏散节点。

**新增模块**：
- 轨道站点和线路数据管理
- 站点容量约束 + 线路输送能力约束
- 公交接驳轨道站点分配优化
- 站点压力评估
- 公交直达 vs 公交-轨道协同方案对比

**轨道交通简化模型**：
```text
站点接纳能力 = 站厅容量 + 站台容量 + 出入口通行能力
线路输送能力 = 列车容量 × 发车频率
站点压力 = 进入站点人数 / 站点最大处理能力
```

**验收标准**：
- 调度结果不超出轨道站点容量
- 可计算站点压力和线路运能利用率
- 三组方案对比输出：公交直达 / 公交接驳轨道 / 混合分流

### 5.5 v0.5.0：完整 Web 可视化

**目标**：完善 Web 演示系统，提供完整的场景配置、仿真运行和结果展示流程。

**新增模块**：

- 仿真运行控制（启动/暂停/重置）

- 多方案对比页

**验收标准**：

## 7. Git 分支管理

```text
main          稳定版本，只放可运行版本（有 tag）
dev           日常开发主分支
feature/*     单个功能开发分支
release/*     发布前整理分支
hotfix/*      紧急修复分支
```

**日常流程**：
1. 从 `dev` 拉出 `feature/*` 分支
2. 功能完成后合并回 `dev`
3. 阶段稳定后拉出 `release/v0.x.0` 分支
4. 测试通过后合并到 `main` 并打 tag
5. 在 `main` 上打 GitHub Release

**tag 命名**：`v0.1.0`、`v0.2.0` 等，遵循语义化版本。

**发布命令**：
```bash
git checkout main
git merge release/v0.4.0
git tag -a v0.4.0 -m "公交-轨道协同疏散版本"
git push origin main
git push origin v0.4.0
```

## 8. 场景版本化

疏散场景使用独立 YAML 文件管理。场景配置**引用数据版本号而非硬编码文件路径**，由系统根据 data_manifest 解析实际文件。

```yaml
# configs/scenarios/scenario_v0.4_bus_rail_001.yaml
scenario_id: scenario_v0.4_bus_rail_001
scenario_name: "公交-轨道协同疏散测试场景"
data_version: data_v0.4                  # 引用 manifest 版本，非文件路径

event:
  type: "flood"
  center: [118.78, 32.05]
  radius_m: 1500
  blocked_roads:
    - road_1023
    - road_2048

demand:
  total_people: 5000
  priority_distribution: "uniform"

bus:
  num_buses: 30
  capacity_per_bus: 60

rail:
  enabled: true

simulation:
  engine: "sumo"
  duration_s: 7200
  step_length_s: 1.0
  random_seed: 42

expected:
  min_completion_rate: 0.80             # 预期最低疏散完成率（用于自动化验证）
  max_avg_waiting_time_s: 900
```

**典型场景类型**：
- 地震导致区域道路封闭
- 大型活动结束后地铁站拥堵
- 暴雨导致部分道路不可通行
- 突发事故导致公交线路绕行

## 11. 环境版本管理

### 11.1 固化的环境文件

| 文件 | 作用 |
| --- | --- |
| `requirements.txt` | Python 依赖（含版本号） |
| `environment.yml` | Conda 环境（推荐用于 SUMO 依赖管理） |
| `docker-compose.yml` | PostGIS + FastAPI + Streamlit + MLflow |
| `.env.example` | 环境变量模板 |

### 11.2 已知兼容性约束

- **SUMO Python 工具**（traci, sumolib）依赖特定 Python 版本，与 OR-Tools 可能存在版本冲突。建议使用 Conda 环境隔离，或通过 `SUMO_HOME` 环境变量引用外部 SUMO 安装。
- **OR-Tools Python 封装**与 Python 3.10/3.11 兼容性最佳，Python 3.12+ 需验证。
- **OpenTripPlanner 是 Java 应用**，引入时将增加 JVM 依赖。建议通过 Docker 容器运行 OTP，避免环境污染。
- **GeoPandas + PostGIS** 的 C 扩展依赖（GEOS, PROJ, GDAL）应在 `environment.yml` 中锁定版本。
- **Streamlit** 与 FastAPI 共存时，需注意端口分配和 CORS 配置。

### 11.3 Docker Compose 服务

```yaml
services:
  postgis:
    image: postgis/postgis:16-3.4
  fastapi:
    build: .
    command: uvicorn src.api.main:app --host 0.0.0.0 --port 8000
  streamlit:
    build: .
    command: streamlit run streamlit_app.py --server.port 8501
  mlflow:
    image: ghcr.io/mlflow/mlflow:v2.14.0
    # 可选，后期引入
```

SUMO 因依赖 GUI 和复杂系统库，建议在本机安装而非容器化，但在环境文件中明确记录版本和安装方式。

## 12. 测试策略

### 12.1 测试分层

| 层次 | 范围 | 工具 | 要求 |

## 13. 版本回滚流程

当新版本出现问题时，回滚步骤：

1. **定位目标版本**：查看 `git tag` 列表，确定要回退到的稳定版本
2. **代码回滚**：`git checkout v0.4.0` 或从该 tag 创建 hotfix 分支
3. **数据回滚**：从 `data/manifest/` 找到对应版本的 `data_manifest.json`，恢复数据文件
4. **场景回滚**：使用对应版本的 `configs/scenarios/` 文件
5. **环境回滚**：checkout 对应版本的 `requirements.txt` 和 `docker-compose.yml`，重建环境
6. **验证**：运行该版本的验收测试，确认指标与原始记录一致
7. **记录**：在 `CHANGELOG.md` 中记录回滚原因和操作

> 核心原则：**每个版本的信息都在 simulation_run 表中有记录**，回滚时查表即可确定所有版本依赖。

## 15. 版本管理原则（终版）

1. 系统代码采用 Git 版本控制，稳定版本通过 Git tag 和 GitHub Release 发布。
2. 每个系统版本对应唯一的代码版本、数据版本、场景版本和实验配置版本，记录在 `simulation_run` 表中。
3. 所有疏散场景采用 YAML 描述，引用数据版本号而非硬编码文件路径。
4. 大型数据文件（> 10MB）通过 DVC 或数据清单管理，不直接提交 Git。
5. 每次仿真实验生成独立编号，保存配置、数据清单、日志、指标和图表。
6. 每个发布版本提供 Release Note，说明变更、版本关联、实验结果和已知问题。
7. 系统运行环境通过 `requirements.txt`、`environment.yml` 和 `docker-compose.yml` 固化。
8. 每个版本必须包含测试，Release 前执行完整的回归测试。
9. 回滚时依据 `simulation_run` 表恢复对应版本的代码、数据、场景和环境。

**一句话总结**：

> 任何一次系统演示、论文实验或版本发布，都必须能够通过 `simulation_run` 表完整追溯其代码、数据、场景、算法参数、仿真环境和实验结果。后期无论是写论文、做答辩、调试系统，还是回退旧版本，都能保持清晰、可靠和可复现。
